# CLAUDE.md · Sparta_AX_Project 작업 헌법

이 문서는 이 프로젝트에서 AI 협업 에이전트가 **매번 지시하지 않아도 항상 따라야 하는 기준**이다. 사용자와 에이전트 모두 이 규칙을 기준으로 일한다. 규칙이 바뀌면 이 문서를 먼저 갱신하고 Decisionlog에 이유를 남긴다.

## 0. 프로젝트 정보
- 프로젝트 이름: Sparta_AX_Project (스파르타 AX 해커톤 제출용)
- 목적: 한 번 끝까지 푼 작업을 **재사용 가능한 Claude Skill로 도구화**하여 제출한다. 핵심 가치는 "한 번 해봤다"가 아니라 "재사용 가능한 도구로 남겼다".
- 주요 사용자: 사내 동료 — `/스킬명` 한 줄로 같은 결과를 다시 뽑아 쓰는 사람
- 최종 산출물: **Claude Skill** = `.claude/skills/<스킬명>/SKILL.md` + 한 번 실행한 결과(입력 → 출력)
- 마감·제약: 해커톤 마감 `확인 필요` / 제출 형식 = `SKILL.md` + 시연(`/스킬명` 1~2분)
- 주제: **미정** (해커톤 주제 확정 후 채움) `확인 필요`
- 현재 사용자 수준: `확인 필요`
- 원하는 진행 방식: `확인 필요` (잠정: 속도·이해 우선)

## 1. 대화와 보고 방식
- 결론을 먼저, 근거·세부는 뒤에.
- 기술 용어는 처음 등장할 때 한 줄로 설명한다.
- 불확실하면 단정하지 말고 `확인 필요`로 표기한다.
- 실행/확인하지 않은 것을 완료라고 말하지 않는다.

## 2. 작업 시작 전 확인
- 현재 폴더 구조와 기존 파일을 먼저 확인한다.
- 기존 파일·변경사항을 함부로 덮어쓰지 않는다.
- 바로 구현하지 말고, 영향 범위와 계획을 먼저 정리한다.

## 3. 파일 구조 원칙 (스킬 제출형으로 조정)
- 이 프로젝트의 산출물은 **스킬 한 개**라서 일반 코드 트리(`src/`·`tests/`·`tools/`)는 기본적으로 만들지 않는다.
- `.claude/skills/<스킬명>/SKILL.md` — 실제 제출물(스킬 본문). 현재 임시 이름 `skill-wip`.
- 스킬이 보조 스크립트를 동반하면 `.claude/skills/<스킬명>/scripts/`에 둔다.
- `docs/` 기획·기준계약·구현·검증 문서 · `docs/references/` 외부 자료(제출 가이드 등).
- `out/demo/` 제출용 실행 결과(입력 → 출력). Git 제외.
- 필요 없는 폴더는 만들지 않고 이유를 남긴다.

## 4. 구현 원칙
- 처음부터 전체를 달리지 않고 가장 작은 성공 단위부터 만든다.
- 각 단계는 다음을 명확히 한다: 왜 필요한가 / 이전 단계와 연결 / 다음 단계로 전달 / 완료 조건 / 검증 방법 / 실패 시 중단점.
- 단계 완료 보고에 포함: 전체 중 현재 위치, 이번에 한 일, 변경 파일, 검증 결과, 남은 단계, 예상 잔여량, 발생한 문제·T-ID, 필요 시 Git commit/push 여부.

## 5. 기준 계약 문서 원칙
- 구현 전 `docs/requirements-contract.md`를 작성한다: 스킬 트리거 키워드 / 입력 형식 / 출력 형식 / 절차(체크리스트) / 완료 기준 / 안전장치(되돌리기 어려운 작업의 자동 실행 차단).
- 같은 기준이 파일마다 다르게 정의되지 않게 한다. 기준이 바뀌면 코드(SKILL.md)보다 계약 문서를 먼저 갱신하고 이유를 기록한다.

## 6. 테스트와 검증 원칙
- 스킬은 **실제로 `/스킬명`을 불러 입력 → 출력이 재현되는지**로 검증한다. 입출력 예시가 SKILL.md의 예시와 일치하는지 확인한다.
- 검증하지 못한 부분은 숨기지 말고 `미검증 범위`로 보고한다. 실패한 시연을 숨기지 않고 원인을 수정한다.

## 7. 보안·개인정보·비밀정보 원칙
- 다음을 코드·문서·로그·테스트 데이터에 넣지 않는다: API 키, 비밀번호, 액세스 토큰, 세션 쿠키, 실제 고객정보, 실제 계좌정보, 민감한 내부 정보.
- 원본 데이터는 수정하지 않는다. 필요하면 마스킹/정규화된 파생 파일을 별도로 만든다.
- **되돌리기 어려운 작업(메일 발송·파일 삭제 등)이 든 스킬은 자동 실행을 막고 `/스킬명`으로만 실행되게 한다.** (제출 가이드 권고)
- 비밀정보가 발견되면 임의로 로그를 편집하지 말고, 노출 범위와 안전한 교체·폐기 방법을 사용자에게 보고한다.

## 8. 문서화 원칙
- README·SKILL.md에는 실제 구현된 기능만 쓴다. 구현되지 않은 기능을 제공한다고 표현하지 않는다.
- 기획서·기준 계약 문서·SKILL.md·시연 결과·README·Worklog·Decisionlog가 서로 모순되지 않게 관리한다.

## 9. 외부 자료 사용 원칙
- 외부 자료 사용 시 기록한다: URL / 제목 / 발행일 또는 확인일 / 가져온 판단 근거 / 적용 범위 / (필요 시) 적용하지 않은 이유.
- 공개 자료로 확인할 수 없는 사실은 단정하지 않는다.

## 10. Git 원칙
- 주요 단계가 끝날 때마다: 변경 파일 확인 → 불필요한 파일 제외 확인 → 검증(시연) 실행 → commit → push.
- 로그, 비밀정보, 로컬 실행 결과, 임시 산출물(`out/`)은 기본적으로 Git에 올리지 않는다.
- 현재 git 저장소 아님 → 필요 시 `git init`부터.

## 11. 로그 기록 규칙

### Worklog.md — 주요 요청마다
모든 주요 사용자 요청이 끝날 때마다 기록한다. 설명 작업이라도 방향·구조·판단에 영향을 주면 기록한다.
```
### W-001 · 작업 제목
**요청** / **수행 작업** / **변경 파일** / **검증**(미실행 시 사유) / **판단 근거** / **결과**(완료·남은 작업)
```

### Decisionlog.md — 방향을 바꾸는 결정만
대상: 문제 선정, 기술 스택 선택, 구현 범위 확정·축소, 데이터 처리 방식, 보안·개인정보 처리, 테스트·검증 방식, 배포·제출·공유 방식, 큰 구조 변경. 사소한 수정은 Worklog만.
```
### D-001 · 결정 제목
**상황** / **검토한 선택지** / **결정** / **근거** / **영향** / **재검토 조건**
```

### Troubleshootinglog.md — 실제 문제 발생 시
실제 오류·실패·환경 문제·검증 실패·설계 충돌이 발생하면 기록한다. 같은 문제가 반복되면 새 T-ID를 만들기 전에 기존 T-ID를 먼저 확인한다.
```
### T-001 · 문제 제목
**발생 상황** / **증상** / **확인된 원인**(+불확실점) / **조치**(+최종 해결) / **재발 방지**
```

## 12. 완료 보고 기준
작업 완료 시 다음 형식으로 보고한다. 실행하지 않은 검증은 완료라고 하지 않는다.
```
완료했습니다.
- 수행한 작업:
- 변경 파일:
- 검증 결과:
- 기록: Worklog / Decisionlog / Troubleshootinglog:
- 남은 작업:
- 한계 또는 미검증 범위:
- Git 상태:
```
