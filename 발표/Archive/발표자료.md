# 키즐링 박람회 리드 후속 이메일 자동화

## 담당자 1명이 8시간 쓰던 후속 이메일을, "명단만 넣으면 나오는" 자동 초안으로 바꿨습니다

> 이 자료는 오직 "귀사가 낸 이 문제를 어떻게 풀었나"만 다룹니다. 경력·자기소개는 이력서에 있어 전부 뺐습니다.
> 실제 발송은 하지 않습니다. 사람이 검토한 뒤 보냅니다 — 그게 이 문제의 정답 범위입니다.

## 슬라이드 1 · 한 장 요약 (Executive Summary)

### 받은 문제 (귀사 발제 원문)

> "박람회 하나가 끝나면 리드가 50명씩 쌓입니다. 후속 이메일을 리드 1명당 10~15분씩 쓰면 50명이면 8시간 이상 — **1인이 감당 불가**입니다."
>
> "원장은 '원생 효과', 팀장은 '도입이 쉬운가', 사업개발은 '수익 모델'을 묻는데, **같은 톤의 이메일은 누구에게도 꽂히지 않습니다.**"

<sub>※ 수치(50명·10~15분·8시간)는 저희가 실측한 값이 아니라 **귀사가 제시한 시나리오·업계 관행 기준값**입니다.</sub>

### 만든 것 (한 문장)

**박람회 리드 명단을 넣으면 → 직책별로 맞춤화된 후속 이메일 초안 + 사람이 꼭 봐야 할 리드 표시 + 후속 캠페인 기획안 + 한눈 요약 화면까지 자동으로 나옵니다.** 발송 직전 단계까지를 대신하고, 보내는 결정은 사람이 합니다.

### 핵심 임팩트 (실제 산출물 기준, 직접 재확인)

| 무엇을 | 결과 |
|---|---|
| **리드 50건 → 후속 이메일 초안** | **49건 자동 생성** (내용이 똑같은 완전 중복 1건은 설계한 규칙에 따라 걸러 통합) |
| **직책이 다르면 제안도 다르게** | 원장·팀장·사업개발 **행동 요청 3종을 실제로 다르게** 작성 |
| **자동화가 단정하면 안 되는 리드** | **5건을 "사람 검토 필요"로 자동 분리** (이메일 없음·상담 기록 비어있음·내용 모호) |
| **다음 액션 제안** | 리드 패턴 기반 **후속 캠페인 기획안 2건** 자동 작성 |

> **AI가 대신한 일**: 리드 49명분 맞춤 문안을 한 번에 작성하고, 손대야 할 예외 리드를 골라냄.
> **사람이 내린 판단**: "직책별로 무엇이 계약을 만드는가"(원장=원생 효과 / 팀장=도입 난이도 / 사업개발=수익 구조), 급한 리드일수록 즉시성을 강하게(상=이번 주 내 즉시 행동) 덜 급한 리드는 부드럽게 하는 재촉 강도 설계, 사람이 꼭 봐야 할 리드의 기준, 그리고 **발송은 반드시 사람이 하도록 한 안전장치**.

**이 한 장의 결론:** 귀사의 8시간짜리 수작업 병목을, 직책을 구분해 초안을 뽑고 위험한 리드는 사람에게 넘기는 자동화로 바꿨습니다. 완벽하다고 말하지 않고, **어디까지 자동화하고 어디서 사람이 판단하는지**를 설계한 결과를 보여 드립니다.

## 슬라이드 2 · 문제 이해

### 기업이 낸 문제 (원문 그대로)

> "박람회 하나가 끝나면 리드가 50명씩 쌓입니다. 후속 이메일을 리드 1명당 10~15분씩 쓰면 50명이면 8시간 이상 — 1인이 감당 불가입니다."

키즐링 영업팀은 교육 박람회(에듀플러스위크 등)에서 명함과 상담 메모를 수십 건씩 모읍니다. 그런데 **후속 이메일은 담당자 한 사람이 전부 손으로 씁니다.** 이 수작업이 병목입니다.

*(수치는 실측이 아니라 문제를 낸 회사가 제시한 시나리오·업계 관행 기준입니다.)*

### 제가 다시 정의한 진짜 문제

원문은 "이메일 쓰기가 힘들다"지만, 실제 아픈 곳은 네 군데입니다. **누가·언제 아픈지**로 다시 씁니다.

| 진짜 아픈 곳 | 누가·언제 | 왜 치명적인가 |
|---|---|---|
| **① 시간 폭발** | 담당자 1인, 박람회 직후 | 1명당 10~15분 × 50명 = **8시간+**, 한 사람으로 감당 불가 |
| **② 대상별 말이 안 통함** | 원장·팀장·사업개발 각각 | 직책마다 궁금한 게 다른데 같은 톤은 **누구에게도 안 꽂힘** |
| **③ 대화가 증발** | 부스에서 나눈 상담 | 이메일에 안 담기면 "우리를 기억 못 하는 회사" 인상 |
| **④ 48시간 골든타임 이탈** | 박람회 후 이틀 | 지나면 관심이 급격히 식고 경쟁사로 넘어감 |

**핵심은 '빨리 쓰기'가 아니라 '48시간 안에, 상대 직책에 맞는 말로, 부스 대화를 기억한 채' 쓰는 것.** 이 세 조건이 동시에 걸려서 사람 손으로는 물리적으로 불가능합니다.

### 문제의 크기 (Before)

- **1명당 10~15분** → 50명이면 **8시간 이상** *(회사 제시 시나리오 기준)*
- **박람회 리드의 70~80%가 후속 관리 부재로 소멸** *(업계 통계, 제공 자료 인용 — 이 프로젝트 실측 아님)*
- **골든타임은 단 48시간** — 이 창을 놓치는 순간 대부분의 리드가 식습니다.

### 이 문제를 왜 이렇게 봤는가 — 도메인 이해

- **키즐링**은 알파세대(2010년대 이후 출생) 어린이용 안전 숏폼 챌린지 + 학습 플랫폼을 만드는 에듀테크 회사입니다.
- B2B 핵심 상품 **키즐콘**은 학교·학원용 **온라인 대회 접수·결제·심사·결과발표를 한 번에** 처리해 주는 서비스.
- 박람회 리드는 **직책 3종**으로 갈리고, 각자 사는 이유가 다릅니다:
  - **원장** — 단독 결정 / 원생 재능·학부모 만족을 봄
  - **팀장** — 실무 검토 / 도입 난이도·운영 공수 절감을 봄
  - **사업개발** — 계약 설계 / 수익 모델·공급 확장성을 봄

**같은 상품이라도 직책마다 '전환 스위치'가 다르다** — 이 지점이 문제의 본질이자, 뒤에서 제가 직접 설계한 판단의 출발점입니다.

> **입력 → 출력 한 줄 미리보기:** L001(한빛영어학원 원장, 상담메모="대회 수기 운영이 번거로움") → "[키즐링] 원장님, 아이들의 재능을 보여주는 안내" 제목 + 원생 효과 소구 + 무료 데모 요청 초안이 자동 생성 *(전체 사이클은 슬라이드 4에서 그대로 제시)*.

## 슬라이드 3 · 해결 접근 & 의사결정

### 한 줄 요약

> **"같은 톤 이메일은 누구에게도 안 꽂힌다"** — 그래서 리드 명단을 넣으면 직책별로 다르게 말을 거는 후속 이메일 초안을 자동으로 써 주되, **발송 버튼은 사람이 누른다.**

### 접근 가설 — 왜 이 방식인가

문제의 핵심은 "속도"가 아니라 **"직책별로 다른 말을 거는 것을 사람이 손으로는 못 한다"**는 데 있었습니다. 원장은 "우리 원생에게 뭐가 좋아지나", 팀장은 "우리 시스템에 붙이기 쉬운가", 사업개발은 "이걸로 돈이 되는 구조인가"를 봅니다. 같은 이메일을 세 사람 모두에게 보내면 셋 다에게 안 꽂힙니다.

그래서 **리드 명단을 넣으면 → 직책과 부스 상담 내용을 읽고 → 그 사람에게 맞는 초안을 대신 써 주는 것**을 만들기로 했습니다. 단 실제 발송은 하지 않습니다. 과제가 정한 범위가 "초안 생성까지"이고, 현장에서 급히 받은 명함·메모는 빈칸·모호가 흔해 사람이 마지막에 봐야 하기 때문입니다.

### 버린 대안과 버린 이유

| 검토한 대안 | 버린 이유 |
|---|---|
| **모두에게 같은 행동 요청** | "직책별 차별화"라는 채용사 평가 기준을 정면으로 못 맞춤. 누구에게도 안 꽂힘 |
| **상담 메모가 빈 리드도 그럴듯하게 지어내기** | 없는 대화를 지어내면 신뢰가 무너짐 → 지어내지 않고 일반 문구 + 사람 검토로 대체 |
| **점수를 새로 매겨 우선순위 재계산** | 과제 금지 범위(리드 점수화). 이미 있는 우선순위 값을 그대로 쓰고, **압박 강도만** 조절 |
| **완전 자동 발송** | 데이터 품질이 들쭉날쭉 → 사람이 마지막에 검토하는 안전장치가 필수 |

### 핵심 흐름 한 장 (사람이 알아보는 동작으로)

```
① 리드 명단 읽기
        ↓
② 직책·부스 상담 내용 파악   (원장? 팀장? 사업개발? / 부스에서 무슨 얘기 나눴나)
        ↓
③ 직책별 맞춤 초안 작성       (제목·본문·다음 행동 제안을 직책마다 다르게)
        ↓
④ 사람이 꼭 봐야 할 리드 표시  (이메일 없음·상담 메모 빈칸·모호·중복 → 검토 표시)
        ↓
⑤ 후속 캠페인 제안           (비슷한 리드를 묶어 "다음 2주 무엇을 보낼지" 기획안)
```

**포인트: ③과 ④는 별개다.** 잘 만든 초안(③)만 내놓는 게 아니라, **"이건 사람이 반드시 확인하세요"라고 스스로 표시(④)**합니다. 이게 자동화가 아니라 **신뢰할 수 있는** 자동화의 차이입니다.

### AI가 대신한 일 vs 내가 내린 판단 (AX 핵심)

| **AI가 대신한 일** (양·속도) | **내가 내린 판단** (전략·안전) |
|---|---|
| 리드 하나하나 개인화 문안 대량 작성 | **무엇이 전환을 만드는가** — 직책별로 무엇에 반응하는지, 어떤 다음 행동을 제안할지 매핑 |
| 명단 전체의 직책·기관·관심사 분포 분석 | **얼마나 세게 밀지** — 상=즉시 행동, 중=일정 조율, 하=천천히 관계 유지 |
| 이상한 리드(빈칸·모호·중복) 후보 찾아내기 | **사람이 꼭 봐야 할 기준(검토 표시)**을 직접 정의 |
| — | **중복·빈칸을 어떻게 처리할지 원칙**, 그리고 **"발송은 사람이"** 라는 안전장치 |

한 줄로: **AI는 "많이, 빠르게"를 맡고, 사람은 "무엇이 옳은가"를 정했습니다.** 그리고 그 경계선에 사람 검토를 남겨 두었습니다.

> **★ 막혔던 지점 & 돌파 ★** 만든 것이 새 명단에도 똑같이 도는지 확인하려고, **같은 도구를 완전히 새로운 가상 리드 명단으로 처음부터 다시 돌려봤습니다.** 기본 규격(글자 수·필수 항목·중복 제거 등)은 전부 통과했지만, 급한 리드용 재촉 문구가 덜 급한 리드 2건에 잘못 섞이는 미세 편차를 발견했습니다. 이 자동화는 정해진 버튼만 누르는 기계가 아니라 **매뉴얼을 읽고 그때그때 판단하는 담당자에 가까워서**, 판단 경계('중'과 '상' 사이)에서 미세하게 흔들릴 수 있기 때문입니다. **원본 제출물에는 이 편차가 없었고**(같은 조건 리드 전부 규칙 준수), 애초에 이 흔들림을 잡으려고 ④번 "사람이 꼭 봐야 할 리드 표시"와 "발송 전 사람 검토"를 흐름 안에 넣어둔 것이 핵심입니다. *이 흔들림이 매번 나는지 어쩌다 한 번인지는 1회 실행만으로 단정할 수 없어(추정) 개선 과제로 남겨 두었습니다.*

## 슬라이드 4 · 산출물 & 실증 — "이런 리드가 들어오면, 이런 이메일이 나옵니다"

### 데모 — 실제 리드 1건이 통과하는 한 사이클 (리드 L001)

> **[녹화 권장]** 화면 녹화 영상은 아직 없습니다. (녹화 예정 내용: 명단 파일을 넣는 순간부터 직책별 초안이 나오고 검토 표시가 붙기까지 한 사이클.) 아래는 조작 없이 실제 데이터 **리드 L001**의 입력 → 출력을 그대로 옮긴 것입니다.

**① 들어간 것 — 부스에서 받은 명함 한 장 수준의 정보**

| 항목 | 내용 |
|---|---|
| 이름·직책 | 김민준 **원장** |
| 기관 | 한빛영어학원 (학원) |
| 관심 | 대회운영 |
| 후속 우선순위 | **상** (지금 바로 움직여야 하는 리드) |
| 부스 상담 메모 | "원생 180명 규모 영어학원. 자체 영어스피치 대회를 매년 여는데 접수·심사·결과발표를 **전부 수기로 해서 힘들다**고 함. 원생 재능을 학부모에게 보여줄 수단을 찾고 있음." |

**② 나온 것 — 그대로 발송 검토에 올릴 수 있는 초안 (손대지 않은 실제 생성 전문)**

> **제목:** [키즐링] 김민준 원장님, 우리 원 아이들의 재능을 보여주는 키즐콘 안내
>
> 김민준 원장님, 안녕하세요. 에듀플러스위크에서 인사드린 키즐링입니다. 한빛영어학원의 영어스피치 대회를 접수·심사·결과발표까지 수기로 운영하시느라 번거롭다고 하신 말씀이 기억에 남습니다. 키즐콘은 대회 접수·결제·심사·결과발표를 한 번에 자동화해 그 수고를 덜고, 아이들의 도전을 영상으로 남겨 원생의 성취를 학부모께 선명하게 전해 드립니다. 원생 재능이 눈에 보이면 학부모 만족과 재등록으로 이어집니다. **효과를 먼저 보시도록 이번 주 안에 무료 데모 일정을 제안드립니다.**
>
> **사람 검토 필요:** 아니오 (바로 발송 검토 가능)

**③ 이 한 통이 자동으로 지킨 4가지**

- **맞춤 인사** — 이름·직책·박람회명으로 "당신을 기억한다"
- **부스 대화 그대로 소환** — "수기로 힘들다"는 그날의 말을 다시 꺼냄 → *상담 내용 휘발* 문제 해결
- **직책에 꽂히는 소구점** — 원장이 궁금한 건 기능이 아니라 **원생 효과·학부모 만족**. 그 언어로 말함
- **행동 1개** — 우선순위가 '상'이라 **"이번 주 안에"** 즉시 행동을 요청 (48시간 골든타임 대응)

### 핵심 결과물 — 리드 명단 한 장이 만들어 낸 5가지

| 결과물 | 무엇을 대신해 주는가 |
|---|---|
| **개인화 이메일 초안 49통** | 담당자 1인이 손으로 매달릴 후속 문안을, 리드마다 다르게 자동 작성 |
| **사람이 볼 리드 표시 5건** | "이건 기계가 단정하면 안 된다"를 골라내 사람에게 넘김 |
| **직책별로 다른 3종의 다음 행동** | 원장·팀장·사업개발에게 각각 다른 제안 |
| **후속 캠페인 기획안 2건** | 리드 한 통을 넘어, 어떤 그룹을 어떤 순서로 공략할지 |
| **한눈 요약 화면** | 위 결과를 바쁜 사람이 훑어볼 수 있게 정리 |

**캠페인 기획안, 넣으면 이렇게 나옵니다 (한 줄 예시):**
- **팀장 그룹 12명 (대회운영 자동화)** → Day 0 도입 난이도 초안 발송 / Day 2~3 연동 체크리스트 / Day 7 운영 공수 절감 미팅
- **사업개발 그룹 12명, 중복 제외 실질 11명 (파트너십)** → Day 0 파트너십 초안 발송 / Day 2~3 수익 모델 1페이지 제안서 / Day 7 수익 배분·공동 마케팅 협의 미팅

**숫자로 본 산출물 (원본 50건 기준, 실제 산출물에서 직접 세어 확인):**

- **초안 49통** — 원본 50건 중 완전히 똑같은 중복 1건(L050)을 대표 리드로 합쳐 제외
- **본문 250~279자** — 과제 규격(250~400자) 충족, 모든 통이 길이 미달·초과 없음
- **직책별 다음 행동 3종** — 원장 = 무료 데모 신청 / 팀장 = 기술 도입 미팅 / 사업개발 = 파트너십 제안서 송부. 제목도 직책마다 다름 *(본문 문구 기준 17/17/10건, 제목까지 포함하면 18/18/10건 — 일부 제목이 행동 요청 문구를 함께 담음)*
- **사람이 볼 리드 5건** — 아래 참조
- **후속 캠페인 기획안 2건** — 팀장 대상 / 사업개발 대상

> 참고: 직책 인원(원장 20·팀장 18·사업개발 12)과 행동 문구 출현 수가 1:1로 맞지 않는 건, 사람 검토 대상 리드에는 발송용 행동 문구를 일부러 넣지 않았고 중복을 제외했기 때문입니다. 인원수와 등치하지 않도록 표기했습니다.

### "기계가 단정하면 안 되는 리드"는 사람에게 넘깁니다 (실제 5건)

같은 자동 흐름이 스스로 **"이건 내가 판단하면 안 된다"**고 표시한 리드입니다. 이게 이 결과물이 신뢰받는 이유입니다.

| 리드 | 문제 | 자동 처리 방식 |
|---|---|---|
| **L009** | 이메일 주소 없음 | 영업 문구 대신 **연락처 확인용 후속 메모**로 전환. 제목에 **[검수 필요]** 명시 → 잘못 발송 방지 |
| **L012·L015·L046** | 부스 상담 메모 공란 | 상담 내용을 **지어내지 않고** 일반 문구로 대체, "발송 전 담당자 보강 필요" 표시 |
| **L033** | 상담 메모 모호 | 니즈가 불명확 → 사람이 확인 후 발송하도록 표시 |

**핵심:** 정보가 부족할 때 그럴듯하게 지어내는 대신, 멈추고 사람을 부릅니다. "우리를 기억 못 하는 회사" 인상을 주는 것보다, 담당자가 30초 보강하는 편이 낫다는 판단입니다.

## 슬라이드 5 · 임팩트 — 이 문제를 풀면 무엇이 달라지는가

### Before → After

| 항목 | Before (현행 수작업) | After (이번 결과물) |
|---|---|---|
| 리드 1명당 후속 작성 | 10~15분 *(과제 제시 시나리오 기준, 실측 아님)* | 초안이 이미 완성된 상태로 도착 |
| 리드 50명 전체 | **8시간 이상, 1인 감당 불가** *(과제 제시 시나리오 기준)* | 명단을 넣으면 초안 일괄 생성 |
| 직책별 맞춤 | 같은 톤 → 누구에게도 안 꽂힘 | **직책 3종 차별화** (원장·팀장·사업개발 각각 제목·소구점·다음 행동이 다름) |
| 상담 내용 반영 | 자주 휘발 → "기억 못 하는 회사" | 부스에서 나눈 대화를 초안 첫 문단에 그대로 인용 |
| 사람이 봐야 할 리드 | 눈으로 일일이 골라야 함 | **위험 리드 5건 자동 표시**(이메일 없음·상담메모 빈칸·모호) |

**측정 방식과 가정 (정직하게):**
- Before의 `10~15분`, `8시간+`, `48시간 골든타임`은 **실측이 아니라 귀사가 제시한 시나리오·업계 관행 기준값**입니다. 저희가 잰 값이 아닙니다.
- After의 산출 수치(아래)는 **실제 결과물을 전수 재계산한 측정치**입니다.
- **After의 "몇 % 시간 절감"은 측정하지 않았으므로 수치를 만들지 않았습니다.** 대비는 "손으로 다 쓰던 후속 → 초안 자동 완성"이라는 정성 대비까지가 정직한 선입니다.

### 실제 산출물 (원본 50건, 측정치)

| 지표 | 값 |
|---|---|
| 완성된 이메일 초안 | **49건** (내용이 똑같은 중복 1건은 대표 1건으로 통합) |
| 본문 길이 | **250~279자** (규격 250~400자 충족, 전수 재계산 min 250 / max 279) |
| 직책별 다음 행동 | **3종 차별화** — 원장=무료 데모 / 팀장=기술 도입 미팅 / 사업개발=파트너십 제안서 |
| 사람 검토 표시 | **5건** (이메일 없음 1 · 상담메모 빈칸 3 · 상담메모 모호 1) |
| 후속 캠페인 기획안 | **2건** 자동 생성 |

### 정성 효과 — 왜 이게 "전환"으로 이어지나

- **직책별 차별화 →** 원장에겐 "원생 효과", 팀장에겐 "도입이 쉬운가", 사업개발에겐 "수익 구조". 한 사람 한 사람이 궁금해하는 딱 그 지점을 건드립니다.
- **상담 내용 재현 → "기억하는 회사" 인상.** 예: 원장님 리드 초안 첫 문단이 *"영어스피치 대회를 접수·심사·결과발표까지 수기로 운영하시느라 번거롭다고 하신 말씀이 기억에 남습니다."* — 부스 대화를 그대로 되살립니다.
- **48시간 골든타임 대응.** 관심이 뜨거운 '상' 우선순위 리드는 "이번 주 안에" 같은 즉시 행동을, '중·하'는 부담 없는 자료 안내로 강도를 낮춥니다.
- **사람 검토로 리스크 제거.** 상담 기록이 비었으면 내용을 **지어내지 않고** 사람에게 넘깁니다. 이메일이 없으면 발송 대신 "전화·방문으로 연락처부터 확인" 메모로 전환합니다.

> 참고(귀사 제공 자료 기준, 저희 실측 아님): 개인화 후속이 없으면 박람회 리드의 **70~80%가 그냥 사라진다**고 합니다. 상담 내용을 언급하면 응답률이 달라진다는 것도 같은 자료의 벤치마크입니다.

## 슬라이드 6 · 적용 가능성 & 한계 & 비전

### 우리 환경에 붙일 때 필요한 것: 새 명단 하나

- 이번 박람회 명단이든, 다음 박람회 명단이든 **같은 형식의 리드 표(이름·기관·직책·상담 메모·관심 기능·우선순위)** 하나만 넣으면 됩니다.
- 그러면 자동으로: **정리 → 직책별로 나누기 → 개인화 초안 쓰기 → 사람이 봐야 할 리드 표시**까지 이어집니다.
- 담당자가 배울 새 프로그램도, 발급받을 외부 키도, 설치할 도구도 없습니다. 직책별 소구점·우선순위별 문구 강도·"이건 사람이 봐야 함" 판단 기준이 **이미 이 흐름 안에 규칙으로 박혀 있어서**, 명단만 바꿔도 같은 품질로 재현됩니다.

> **실제로 확인함:** 이 저장소 밖 완전히 다른 폴더에서, 처음 보는 가상 리드 명단으로 전체를 처음부터 다시 돌렸습니다. 하드 요건(필수 4요소·본문 길이·중복 제외·캠페인 근거 리드)은 통과했고 — **같은 실행에서 문구 2건이 흔들려, 사람 검토 필요성을 오히려 다시 확인했습니다.**

### 정직하게, 아쉬웠던 점 (완벽한 척은 하지 않습니다)

**① 문구가 실행마다 미세하게 흔들린 사례 2건 — 그래서 사람이 마지막에 봅니다**
- 다른 폴더에서 새 명단으로 다시 돌렸을 때, 우선순위 '중'인 원장 리드 2건에 상급 리드용 긴급 문구가 잘못 섞이거나 기본 문구가 통째로 빠지는 편차가 나왔습니다.
- 단, **제출한 원본 명단에서는 같은 조건 리드가 전부 규칙대로 나왔습니다.** 편차가 난 건 다른 폴더 재실행분(원본과 다른 별도 명단)이며, 원본 산출물의 결함이 아닙니다.
- 원인 추정: AI가 글로 된 규칙을 따르다 보니 경계선에서 판단이 실행마다 조금씩 달라질 수 있습니다. **바로 이것이 "AI가 초안, 사람이 최종 검토"인 이유이자, 다음에 손볼 지점입니다.**

**② 노션·구글 시트 공개 링크는 아직 안 올렸습니다**
- 결과물(초안 표)은 다 만들어져 있고 올리는 절차도 문서로 정리해 뒀지만, **실제 공개 URL 생성은 이 작업 안에서 마치지 못했습니다.** "제출 완료"라고 말하지 않겠습니다.

### 이건 아쉬움이 아니라, 처음부터 선을 그은 범위입니다

- **실제 메일 발송·CRM 연동·다국어·리드 점수 매기기**는 **이 과제가 처음부터 선을 그은 범위 밖**입니다. "못 한 것"이 아니라 "하면 안 되는 것"입니다.
- 이 흐름은 "사람이 검토 후 발송"까지가 설계 의도입니다. 초안까지 자동, 발송은 사람 — 이게 안전장치입니다.

### 다음 스텝 (작업 중 발견해 내부 점검 메모에 남긴 개선 과제 기준)

1. **문구 규칙 자동 점검 추가** — 사람 검토 없이 나가는 리드는 직책별 핵심 문구(무료 데모 / 기술 도입 미팅 / 파트너십 제안서)가 본문에 실제로 들어갔는지 **한 번 더 자동으로 걸러내기**. → 위 ①번 흔들림 재발 방지.
2. **발송 전 승인 흐름** — 초안 → 담당자 승인 → 발송으로 이어지는 사람 확인 단계를 명시적으로 붙이기.
3. **성과 측정 자리 만들기** — 오픈·응답 결과를 되받아 어떤 직책·문구가 실제로 전환됐는지 다음 명단에 반영.

*이 3가지는 작업 중 발견해 내부 점검 메모에 남긴 개선 후보이며, "먼저 이것부터"라는 우선순위까지 이미 정해져 있습니다.*

## 슬라이드 7 · 마무리

> **이 문제를 계속 맡는다면 —**
> 담당자 1인이 손으로 붙들던 후속 작업(과제 시나리오 기준 리드 50명이면 8시간 넘게 걸리는 일)을, **명단을 넣으면 직책별 초안이 나오고 사람은 판단만 하는 흐름**으로 바꾸고, 실행할수록 문구 규칙과 성과 데이터가 쌓여 스스로 정교해지는 자산으로 키우겠습니다.

> **이 결과물이 보여주는 것 —**
> AI 전환의 핵심은 "AI로 많이 만들기"가 아니라 **"AI가 단정하면 안 되는 선을 사람이 어디에 긋느냐"**입니다. 이 결과물은 그 선을 직책별 소구점 설계·우선순위별 강도·사람 검토 기준으로 그어 보였습니다. 흔들린 2건을 숨기지 않고 사람 검토의 근거로 삼은 것 — 그게 "초안까지는 AI, 발송 판단은 사람"이라는 이 설계가 실제로 작동한다는 증거입니다.

# (별첨) 기술 & 논리 상세

> 이 별첨은 파이프라인의 내부 구조·규칙·검증 수치를 깊이 확인하려는 담당자용입니다. **본 섹션에 한해** 구현 용어(스킬 체인·CSV 스키마·review_flag 등)를 사용합니다. 모든 수치는 제출물 파일에서 직접 재확인한 값이며, 제공 컨텍스트에서 인용한 값은 별도 표기했습니다.

## 1. 전체 흐름 — 기능별 단계 체인 (각 단계가 받는 것 / 내놓는 것)

수작업 후속을 통째로 자동화하는 대신, **한 사람이 후속 이메일을 쓸 때 실제로 거치는 판단 단계**를 그대로 분리된 단계로 나눴습니다. 각 단계는 앞 단계의 산출물만 입력으로 받고, 다음 단계가 쓸 산출물만 내놓습니다.

| 단계 | 받는 것 (입력) | 하는 일 | 내놓는 것 (출력) |
|---|---|---|---|
| **① 리드 분석**<br>`kizzling-analyze-leads` | 리드 명단 CSV (50행) | 직책·기관·관심기능·우선순위 분포 집계, 이메일 결측·상담메모 결측/모호·완전중복 탐지, `review_flag` 후보 식별 | 리드 분포 + 예외 후보 목록 |
| **② 메시지 전략**<br>`kizzling-message-strategy` | 리드 분포 + 전략 설계 규칙 | 직책별 소구점·제목 패턴·CTA 매핑, 우선순위별 CTA 강도, `review_flag` 기준 확정 | 직책·우선순위별 문안 규칙 세트 |
| **③ 이메일 생성**<br>`kizzling-generate-emails` | 리드 분석 + 전략 규칙 | 리드별 개인화 초안 생성, 예외 리드 분기 처리, 스키마 검증 | `email_drafts.csv` + 예외 요약 |
| **④ 캠페인 기획**<br>`kizzling-campaign-plan` | 초안 + 리드 패턴 | 세그먼트 정의·근거 리드·대표 인용·후속 시퀀스 구성 | 캠페인 기획안 2건 |
| **⑤ 시각화**<br>`kizzling-visualize-campaign` | 캠페인 기획안 | 정적 HTML 한눈 요약 생성 (스크립트·폼·외부자원 없음) | `campaign_plan_preview.html` |
| **⑥ 검수·품질 QA**<br>`kizzling-review-submit` → `kizzling-quality-audit` | 전 산출물 | 요구사항 매핑, 스키마·길이·플래그·중복·근거 검증, 제출 패키징 | 검수 리포트 + 제출 요약 |

- **Standard(재현) 검증 경로**: ① → ② → ③ → ⑥ 만으로도 동일 스키마의 이메일 초안이 다시 생성되고 QA됩니다. ④⑤는 Challenge용 확장 단계입니다.
- **의존성 없음**: 별도 스크립트·외부 assets·외부 API에 의존하지 않습니다. `follow_up_priority` 입력값은 그대로 CTA 강도 조절에만 쓰고, 새 점수를 계산하지 않습니다(별도 리드 스코어링 없음 — 과제 금지 범위).

## 3. 예외 처리 규칙 — review_flag · 중복 · 결측

**핵심 원칙: 자동화가 단정하면 안 되는 리드를 사람에게 넘긴다 (Human-in-the-loop).**

**review_flag = Y (사람 검토 필요) — 총 5건, CSV 직접 확인**

| 사유 | 리드 | 처리 |
|---|---|---|
| 이메일 주소 결측 | L009 | 발송 불가 → 세일즈 CTA 제거, 연락처 확인용 후속 메모로 전환, 제목에 `[검수 필요]` 표기 |
| 상담메모 결측 | L012 · L015 · L046 | 상담 내용을 **지어내지 않고** 일반화 → 발송 전 담당자 보강 |
| 상담메모 모호 | L033 | 구체적 니즈 불명확 → 발송 전 담당자 보강 |

**중복 처리**
- L039 / L050 은 `lead_id`만 다르고 내용 동일한 완전 중복 → **L039를 기준 레코드로 유지, L050을 초안 대상에서 제외**하고 제외 기록을 남김.
- 근거: 중복 발송은 같은 리드에게 두 번 접촉하는 효과 → 브랜드 피로·신뢰 저하.

**데이터 정합 판단 (정직한 포인트)**
- 발제 원문은 상담메모 결측을 **2건**으로 기술하나, 실제 제공 CSV의 결측은 **3건**(L012·L015·L046).
- "산출물은 실제 데이터를 입력으로 처리하므로 실제 데이터 품질을 우선한다"는 이유로 **실제 CSV를 기준으로 채택**. → 발표에서는 "실제 데이터 기준 5건"으로 인용해야 정확합니다.

## 4. 직책별 차별화 + 우선순위별 CTA 강도 규칙

**"같은 톤 이메일은 누구에게도 안 꽂힌다"** — 의사결정 역할별로 전환 동기가 달라 메시지를 직책별로 분기했습니다(설계 근거: `design-rationale.md`).

| 직책 | 전환을 만드는 것 | 제목 방향 | CTA |
|---|---|---|---|
| **원장** | 원생 효과·학부모 만족을 **직접 확인**해야 전환 | "우리 원 아이들의 재능을 보여주는…" | 무료 데모 신청 |
| **팀장** | 실무 병목(운영 공수·연동 난이도) 해소 확인 | "기존 운영에 쉽게 붙는 대회 운영 자동화" | 기술 도입 미팅 |
| **사업개발** | 기능보다 **반복 가능한 수익 구조·유통 확장성** | "키즐콘 공동 공급·파트너십 제안의 건" | 파트너십 제안서 송부 |

**우선순위별 CTA 강도** (입력값 상/중/하를 강도 조절에만 사용):

| 우선순위 | 강도 | 의도 |
|---|---|---|
| 상 | 48시간 내 즉시 행동 (미팅·데모·제안서) | 골든타임 전환 |
| 중 | 일정 조율·자료 안내 | 관심 유지, 과도한 압박 회피 |
| 하 | 소개자료·사례집 중심 장기 너처링 | 부담 낮춰 관계 유지 |

**CTA 분포 (CSV 본문 직접 확인 — 검증치)**: 무료 데모 **17** / 기술 도입 미팅 **17** / 파트너십 제안서 **10** (본문 문구 기준). 제목까지 포함하면 18 / 18 / 10 — 일부 제목이 CTA 동사를 함께 담기 때문입니다.
- 직책 인원(원장 20 / 팀장 18 / 사업개발 12)과 1:1로 등치되지 않습니다. 이유: ① review_flag=Y 리드는 발송 CTA를 부여하지 않음(원장·팀장 각 1건이 여기서 빠짐) ② 중복·이메일 결측 제외. → **"CTA 수 = 직책 인원"으로 읽으면 오류**입니다.

## 5. 재현성 테스트 — 방법과 결과

**목적**: 새 리드 CSV(동일 스키마)에도 같은 품질로 동작하는가?

**방법**: 이 저장소와 **무관한 독립 폴더**에 스킬 체인을 복사하고, **새 더미 리드 16건**(이메일 결측·상담메모 결측/모호·완전중복 각 1건 이상 포함)으로 Basic~Challenge~시각화 **전체 체인을 재실행**.

**결과 — 하드 요건 전부 통과**:
- 스키마 4컬럼 · 본문 250~400자 · review_flag 정확도 · 중복 제외 · 캠페인 근거 리드 실재 · HTML 내 스크립트/폼/외부자원 부재 → **모두 통과**. 스킬 파일 자체에 CTA 매핑·우선순위 전략·48시간 골든타임 개념이 내장돼 있어 context 파일 없이도 구조 요건이 동작함을 확인.

**정직한 한계 — 발견한 편차 2건**:
- 재실행분에서 review_flag=N인 **D009**(원장·중)에 상 등급 전용 긴급 문구("48시간")가 잘못 섞이고, **D012**(원장·중)는 표준 CTA "무료 데모"가 통째로 빠져 일반 안내로 대체됨.
- 단, **D009·D012는 독립 테스트의 더미 데이터 ID**입니다(원본의 L009/L012 아님). **원본 제출물 50건의 동일 조건 원장 리드 11건은 이 규칙을 11/11 준수**함을 대조 확인 → **이번 재실행에서만 나타난 런투런(run-to-run) 편차**이며 원본 산출물의 결함이 아닙니다.
- **원인 추정**: 스크립트가 아닌 LLM(스킬) 기반이라, 프로그램적 강제 없이 서술형 규칙만으로는 실행마다 경계 적용에 미세 편차가 생길 수 있음. 1회 실행이라 **우연한 1회성인지 반복 패턴인지는 미확인(불확실점)**.
- **이것이 곧 "사람이 최종 검토하는 이유"이자 개선 과제**입니다. → 개선 방향(7절).

## 7. 개선 과제 (로그 기록 기반 · 미구현)

로그(T-004 "재발 방지", W-019 "남은 작업")에 남긴 개선 후보이며, 아직 구현 전입니다.

1. **품질 QA에 프로그램적 grep 체크 추가** — "review_flag=N 리드는 직책별 CTA 핵심 키워드(무료 데모 / 기술 도입 미팅 / 파트너십 제안서)가 본문에 반드시 존재"를 강제. 현재 QA는 CTA 차별화를 **3개 직책 템플릿 단위로만** 검사해 행별 편차를 놓침 → 5절 편차 재발 방지.
2. 팀장 제목이 `interest_feature`와 무관하게 고정 템플릿을 쓰는 점 개선(기존부터 존재).
3. 노션/구글 시트 공개 링크 실제 생성·접근성 확인(현재 `public_link_prep.md`에 절차만 준비, **미업로드**).

## 8. 범위 밖 (한계 아님 — 스코프 결정)

실제 이메일 발송 · SMTP/Gmail API · CRM 연동 · 다국어 · 리드 스코어링은 **과제가 명시적으로 정한 제외·금지 범위**입니다. "못 한 것"이 아니라 "하면 안 되는 영역"이며, 초안은 사람이 검토 후 발송합니다.

**미검증 범위(정직 표기)**: ① 공개 링크 접근성 ② 완전히 새로운 독립 세션에서의 전체 스킬 체인 end-to-end 재실행 — 두 항목은 문서에 "미검증"으로 명시돼 있습니다.
